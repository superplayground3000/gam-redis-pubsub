System under test

What the pipeline looks like during the coexistence phase

One Redis stream feeds one forward (source) leg, which stamps every message with a routing subject and publishes it to NATS JetStream. Four independent sink groups — two sharded, one prefix-routed, one catch-all — each subscribe to only their slice of the subject space and write to the region Redis. Each leg and each sink group has its own leader election (2 pods for the groups we kill in §4, 1 for the rest).

Phase 3 · 18 s · 4 gates + 1 finding

Graceful leader termination: no loss — but the hand-off is lease expiry, not a graceful shutdown

While a hot key was receiving a continuous stream of sequenced updates, the current leader pod was deleted gracefully (normal kubectl delete pod, default 30 s grace) — first on the forward leg, then on the shard-a sink group. Known gap (confirmed): no graceful pipeline shutdown on SIGTERM

The elector sidecar has no signal handler and the pods have no preStop hook, so on a graceful termination the OnStoppedLeading path — which would explicitly DELETE the running pipeline and release leadership — never executes. The test followed each victim's elector log through its entire termination: the OnStoppedLeading marker never appeared on either leg (elector_delete_ran=false). Failover therefore always pays the full lease timeout (~6 s of leaderless pipeline per event) even when Kubernetes asked politely. Impact today: latency only, not loss — every gate above passed. Per your instruction this run documents the gap; the natural fix (a SIGTERM handler and/or preStop hook in the elector) is left as a follow-up product change.

Method

How the oracles work — and why two of them had to be redesigned

A test of an at-least-once pipeline has one classic trap: duplicates are legal. Any oracle written as "applies must equal what we sent" will randomly fail on a perfectly healthy system the moment a redelivery lands. The first live run hit exactly that (a benign +2 in one lane, and 34 straggler duplicates from one fault phase bleeding into the next phase's counter), so the oracles were redesigned to be duplicate-proof:

OracleNaive form (rejected)Final form (duplicate-proof)
Isolationper-group apply count == emitted countExact-zero checks only: foreign-shard applies = 0, keys outside the lane's namespace = 0. Counts become "≥ emitted" liveness; the surplus is reported as duplicates.
Zero losscount comparisonKey-by-key terminal-state reconciliation: replay the emitted op log (creates, updates, deletes, renames) and require the region to match exactly — including keys that must be absent after a delete.
Ordering—A poller samples hot keys thousands of times during the chaos; every observed value must be ≥ the previous one (7 052 samples this run).
F-2 "nothing applied while broken"global apply-counter delta == 0Key-scoped region scan (f2 keys = 0 during fault) plus a settle barrier between fault phases, since the global counter can't tell one phase's stragglers from another's applies.

It took three runs to get a fully green board — all three failures were harness defects, not pipeline defects (a bash crash on rename bookkeeping, the two naive oracles above, a monitoring pod dying under load, and a results-file race). The product-side behavior was consistent across all three runs. The test also survived its own adversarial review: a cross-model (Codex) review before the first run caught a cleanup bug that could have left the kind node's NATS firewalled cluster-wide.

Caveats

Known gaps and what this test deliberately does not claim

ItemStatusDetail
Graceful SIGTERM pipeline hand-offGAP (product)Confirmed absent on both legs (§4). Failover works via ~6 s lease expiry; zero loss observed. Fix would be an elector signal handler / preStop hook.
Default keyPattern mismatch trapTRAP (config)The chart default matches {employee:…} (singular) but the writer's real keys use plural tags like {employees:…} — a sharded install left on the default would quarantine everything to sx. The e2e overlay sets the plural pattern explicitly and P1 has a guard that fails if it ever goes missing.
Cross-key reorderingby designOrdering is guaranteed per key, not across keys (documented lab design decision). Not tested, not claimed.
F-2 scopeby designThe region store is wiped when its pod is deleted (emptyDir), so the sink-fault phase reconciles only its own disjoint key range — earlier phases' keys are legitimately gone by then.
Concurrent pull ("shared") consumer modeout of scopeExplicitly unimplemented in the chart; rejected loudly at render time.
